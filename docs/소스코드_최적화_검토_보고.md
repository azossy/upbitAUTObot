## 2. 백엔드

### 2.1 트레이딩 루프 내 User 조회 반복 (개선 권장)

**위치**: `backend/app/trading/engine.py`

**내용**: 루프 한 번 돌 때마다 `Bot` 조회 후, 알림이 필요할 때마다 `User`를 별도 `select`로 조회함. 정지/긴급정지 시에도 매번 `select(User).where(User.id == user_id)` 실행.

**영향**: 루프가 60초 간격이므로 실제 부하는 크지 않음. 다만 같은 세션에서 Bot과 User를 함께 가져오면 쿼리 1회로 줄일 수 있음.

**권장**: (선택) 루프 진입 시 또는 알림 발송 직전에 `User`를 한 번만 조회해 두고 재사용. 또는 `Bot` 조회 시 `joinedload(Bot.user)`로 User를 함께 로드.

---

### 2.2 auth 라우터 내 User 조회 패턴 (참고)

**위치**: `backend/app/routers/auth.py`

**내용**: `select(User).where(User.email == ...)` 형태의 조회가 여러 엔드포인트에 반복됨. 로직마다 필요한 조건이 달라 공통 함수로 묶기엔 다소 제한적임.

**영향**: 각 요청당 1~2회 수준이며, 인덱스만 있다면 성능 문제 없음.

**권장**: 현재 상태 유지. 필요 시 `get_user_by_email(db, email)` 같은 헬퍼를 두어 가독성만 높일 수 있음.

---

### 2.3 이메일 발송: CLI vs app 서비스 (의도적 이중 구현)

**위치**: `backend/baejjangi.py` (`_send_test_email_from_env`) vs `backend/app/services/email_service.py`

**내용**: CLI 테스트 메일은 `.env`를 직접 파싱하고 `smtplib`로만 발송. 앱에서는 `email_service.send_email`을 사용.

**이유**: CLI는 `app` 패키지 없이 동작하도록 설계된 것으로 보임. 앱과 동일한 푸터를 붙이기 위해 푸터 문구가 두 곳에 존재.

**권장**: 푸터 문구는 상수로 한 곳에 두고, CLI에서만 그 상수를 참조하거나 복사해 사용하도록 문서화. (현재는 문구 일치만 유지하면 됨.)

---

### 2.4 시세 API (market) — 단일 httpx 클라이언트

**위치**: `backend/app/routers/market.py`

**내용**: 요청마다 `async with httpx.AsyncClient()`로 새 클라이언트 생성.

**영향**: 요청 수가 많지 않으면 문제 없음. 초당 수십 회 이상이면 클라이언트 재사용이 유리.

**권장**: (선택) 앱 lifespan에서 `AsyncClient`를 생성해 `app.state`에 두고, 해당 라우터에서 재사용.

## 3. Flutter 앱

### 3.1 대시보드 초기 로드 — 순차 API 호출 (개선 권장)

**위치**: `upbit_trading_app/lib/presentation/screens/dashboard_screen.dart` — `_fetch()`

**내용**: `getBotStatus()` → `getPositions()` → `getPnlHistory()` → `getBalance()` → `_fetchTicker()` 를 순차 실행.

**영향**: 체감 로딩 시간이 길어질 수 있음.

**권장**: 서로 의존하지 않는 호출은 `Future.wait`로 병렬화.

```dart
final results = await Future.wait([
  api.getBotStatus(),
  api.getPositions(),
  api.getPnlHistory(days: 30),
  api.getBalance(),
]);
// _fetchTicker() 는 별도로 호출하거나 동일한 Future.wait에 포함
```

---

### 3.2 SharedPreferences.getInstance() 반복 호출 (참고)

**위치**: `api_service.dart` (`loadStoredBaseUrl`, `saveBaseUrl`), `biometric_service.dart`, `settings_screen.dart`

**내용**: 필요할 때마다 `await SharedPreferences.getInstance()` 호출.

**영향**: Flutter 쪽에서 내부적으로 캐시하는 경우가 많아, 실제 비용은 제한적. 다만 같은 화면/플로우에서 여러 번 부르면 인스턴스 한 번만 받아서 재사용하는 편이 깔끔함.

**권장**: (선택) Provider/Riverpod으로 `SharedPreferences` 인스턴스를 한 번만 받아 두고, `api_service` 등에서 해당 인스턴스를 주입해 사용.

---

### 3.3 금액 포맷 함수 단일화 (유지보수)

**위치**: `dashboard_screen.dart` — `_formatKrw`

**내용**: 한국식 천 단위 콤마 포맷을 `_formatKrw`로 구현. 다른 화면(positions, trades 등)에서 비슷한 포맷이 필요하면 중복이 생길 수 있음.

**권장**: (선택) 공용 유틸(예: `lib/utils/format.dart`)에 `String formatKrw(num value)` 등을 두고, 대시보드·포지션·거래 내역 등에서 공통 사용.

---

### 3.4 티커 폴링 간격 (참고)

**위치**: `dashboard_screen.dart` — `Timer.periodic(const Duration(seconds: 60), ...)`

**내용**: 60초마다 시세 갱신. 업비트 공개 API 제한(예: 초당 10회)을 고려하면 무난한 설정.

**권장**: 현재 유지. 필요 시 30초 등으로 조정 가능하나, 불필요한 요청은 줄이는 편이 좋음.

---

## 4. 중복 모듈 여부

- **백엔드**: 이메일 발송이 CLI(`baejjangi.py`)와 `email_service` 두 경로인 것은 의도된 분리로 판단. 그 외 동일 기능의 중복 모듈 없음.
- **Flutter**: API 호출은 `ApiService`에 집중되어 있으며, 인증/상태는 `AuthNotifier`에서 처리. 동일 역할의 서비스/프로바이더가 이중으로 존재하지 않음.
