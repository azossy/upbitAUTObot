# 코인 현물 트레이딩 알고리즘 기획 가이드

트레이딩 엔진은 배짱이의 **가장 중요한 부분**입니다. 많은 연구와 단계적 기획 없이 로직을 넣으면 리스크와 손실이 커질 수 있으므로, **알고리즘을 어떻게 기획·연구할지** 절차를 정리했습니다.

---

## 1. 왜 “연구 → 기획 → 검증” 순서가 필요한가

- **한 번의 잘못된 진입/매각**이 실계정에서 반복되면 손실이 누적됩니다.
- **백테스트·시뮬레이션·페이퍼트레이딩**으로 “가설 → 검증”을 반복한 뒤, 실제 주문 로직을 넣는 것이 안전합니다.
- 기획 단계에서 **진입 조건·매각 우선순위·리스크 한도**를 문서로 고정해 두면, 나중에 코드와 동작을 대조하기 쉽습니다.

---

## 2. 알고리즘 기획 절차 (권장 순서)

| 단계 | 할 일 | 결과물 |
|------|--------|--------|
| **1. 연구** | 지표·논문·사례·시장 특성 조사 | 연구 노트, 참고 자료 목록 |
| **2. 가설 수립** | “이런 조건일 때 진입/청산하면 유리하다”를 문장으로 정의 | 가설 1·2·3 … (진입/매각별) |
| **3. 설계** | 가설을 **조건·단계·우선순위**로 구체화 | 진입/매각 로직 명세 (현재 트레이딩_로직_상세_가이드.md 같은 형태) |
| **4. 백테스트** | 과거 데이터로 시뮬레이션, 수익률·손실률·횟수 측정 | 백테스트 리포트, 파라미터 후보 |
| **5. 페이퍼트레이딩** | 실시간 데이터에 주문만 넣지 않고 시그널만 기록 | 일지, 승률·손익비 요약 |
| **6. 리스크·운영 규칙** | 최대 손실, 분할 비중, 동시 보유 종목 수 등 | 리스크 규칙서 |
| **7. 구현·배포** | 설계서대로 엔진에 로직 구현, 소액으로 실거래 검증 | 코드, 모니터링 체크리스트 |

## 3. 연구 단계 — 무엇을 볼 것인가

### 3.1 시장·상품 이해

- **현물**의 특성: 레버리지 없음, “사서 보유 후 파는” 구조, 숏 없음.
- **코인 시장** 특성: 24시간, 변동성 큼, 유동성·스프레드·슬리피지.
- **업비트** 제약: 호가 단위, 최소 주문 금액, API 제한(초당 10회 등).

### 3.2 지표·기법 연구

- **추세**: EMA, MACD, 정배열/역배열, 골든크로스/데드크로스.
- **강도**: ADX, 거래량(평균 대비).
- **과매수/과매도·눌림**: RSI, 스토캐스틱, 지지/저항.
- **다중 시간프레임**: 1h·4h 정배열 + 1h 진입 타이밍 등 (현재 기획서와 맞춤).

**추천**: 하나의 지표만이 아니라 **조합 + 다중 확인**이 왜 필요한지 논문·블로그·백테스트 사례를 찾아 정리.

### 3.3 리스크·심리

- **손절**이 없을 때 한 번의 큰 손실이 수익을 다 깎아먹는 사례.
- **포지션 사이징**(한 번에 얼마나 넣을지), **동시 보유 종목 수** 상한.
- **과최적화(커브피팅)** 위험: 과거 데이터에만 맞추면 미래에는 오히려 손실.

### 3.4 참고할 수 있는 자료 형태

- 업비트 API 문서, 캔들/호가 데이터 형식.
- 기술적 분석 서적·강의(추세·지표·다중 시간프레임).
- 논문/백테스트: “EMA cross + RSI + volume filter” 등 키워드로 검색.
- **자기만의 연구 노트**(Markdown/Notion): “이 조건에서 이렇게 하면 이런 결과가 났다”를 짧게라도 기록.

## 4. 가설 → 설계로 옮기기

- **가설 예**: “1시간봉 골든크로스 + 4시간봉 정배열 + ADX 25 이상 + 눌림목(RSI 50 이하)일 때만 매수하면, 단순 골든크로스보다 승률이 높다.”
- **설계로 옮기기**:  
  - 1차: 시장 국면 상승 + 슬롯 여유  
  - 2차: 골든크로스 O, 정배열 O, ADX≥25, 거래량 조건 O  
  - 3차: 눌림목(단기 EMA 근처, RSI≤50, 거래량 평균 이하)  
  → “세 단계 모두 통과 시에만 매수”처럼 **if–then**으로 문서화.

현재 프로젝트의 **트레이딩_로직_상세_가이드.md**가 이미 이 역할을 하고 있으므로, 연구 결과는 **가설**로 써 두고, 채택한 것은 **트레이딩_로직_상세_가이드.md**에 “이유”와 함께 반영하는 방식을 권장합니다.

## 5. 검증 도구 (우리 프로젝트 기준)

- **가상 데이터 시뮬레이션**: `backend/scripts/test_trading_logic.py`  
  - 실제 주문 없이, 가상 캔들로 진입/매각 조건이 기획대로 동작하는지 확인.
- **백테스트(추가 권장)**:  
  - 과거 캔들 CSV 또는 API로 수집한 데이터를 넣어, 같은 로직으로 “과거에 적용했을 때” 수익률·최대 낙폭·거래 횟수를 계산.
- **페이퍼트레이딩**:  
  - 실시간으로 시그널만 발생시키고, 주문은 넣지 않고 일지로 기록 → 나중에 “실제로 넣었으면 얼마였을지” 정리.

## 6. 리스크·운영 규칙을 먼저 정해 두기

알고리즘 “기획”에 **손실 한도·운영 규칙**을 포함하는 것이 좋습니다.

- **일일/주간 최대 손실 한도** (예: 원금 대비 -N% 도달 시 당일/당주 거래 중단).
- **한 종목 최대 비중** (예: 총 자산의 20% 초과 금지).
- **동시 보유 종목 수 상한** (이미 기획서의 “슬롯” 개념과 연결 가능).
- **손절/익절 비율** (예: 손절 -2.5%, 익절 +5%/+10%/+15% 구간 — 현재 기획과 동일하게 유지·조정).

이것들을 **별도 문서(예: 리스크_운영_규칙.md)** 에 두고, 엔진은 “이 규칙을 위반하지 않는 범위에서만” 주문하도록 설계하는 것이 좋습니다.

## 7. 정리: “가장 좋은” 기획 방법

1. **연구**  
   시장·지표·사례를 정리하고, “이런 조건에서 진입/청산하는 전략”을 **가설**로 여러 개 써 본다.

2. **설계**  
   채택한 가설을 **단계·조건·우선순위**로 명세화한다. (기존 `트레이딩_로직_상세_가이드.md` 활용·보완.)

3. **검증**  
   가상 데이터 → 백테스트 → 페이퍼트레이딩 순으로, **실제 주문 없이** 동작과 수익성을 확인한다.

4. **리스크 규칙**  
   최대 손실·비중·동시 보유 수 등을 문서로 고정하고, 엔진이 이를 준수하도록 설계한다.

5. **소규모 실거래**  
   검증과 규칙이 안정된 뒤, 소액으로 실거래를 돌려 보며 모니터링하고, 필요 시 로직/파라미터만 조금씩 수정한다.

이 순서를 지키면 “많은 연구가 필요한” 트레이딩 엔진을 **체계적으로** 기획할 수 있고, 나중에 “왜 이렇게 만들었는지”도 문서로 추적할 수 있습니다.

## 8. 연구원 에이전트 최박사 (자료수집·연구·정리·가설제안)

알고리즘 기획의 **기초 작업**(자료수집, 연구, 정리, 가설제안)은 **연구원 에이전트 최박사**에게 맡길 수 있다.

- **규칙**: 모든 인용·수치·사례에 **정확한 출처**를 첨부해야 함.
- **메인 산출물**: `docs/트레이딩기법_연구자료.md` — 논문 형식으로 **수집 자료 정리·출처 명시·관련 기법 연구·최신 트렌드·매매기법 가설·제안**을 한 편의 문서에 계속 업데이트·추가. 작업 내용은 수시로 123에 기록.
- **보조**: `docs/연구_노트/` 폴더에 세부 연구 노트 보관 가능.
- **역할 한계**: 검증·결정·실전은 사용자가 수행.

에이전트 규칙: `.cursor/rules/agent-연구원-담당.mdc` (코드네임: 최박사)
