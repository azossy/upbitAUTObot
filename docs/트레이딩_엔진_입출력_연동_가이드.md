# 트레이딩 엔진 입·출력 명세 및 연동 가이드

**목적**: 트레이딩 엔진을 **바이너리로만 배포**(소스 비공개)할 때, 엔진만 구매·사용하는 측이 **어떤 값을 넣어 주고 어떤 값을 받는지**, 그리고 **어떻게 코딩(연동)해야 하는지**를 정의한 문서.  
**유료 엔진** 배포 시 이 문서를 **공개 연동 규격**으로 제공한다.

**관련 문서**: [트레이딩_엔진_상세_기획서.md](트레이딩_엔진_상세_기획서.md) §7 (엔진 분리·바이너리 배포).

---

## 1. 전제

- **엔진**: 별도 프로세스(컴파일된 실행 파일). 소스는 공개하지 않음.
- **호출 측**: 백엔드·자동매매 서비스·외부 봇 등. HTTP/gRPC/소켓 중 정한 **한 가지 프로토콜**로 엔진에 요청을 보내고 응답을 받음.
- **이 문서**: 엔진 **입력 필드 리스트**·**출력 필드 리스트**·**연동 절차(엔진만 사용할 경우 코딩 방법)** 를 정의. 엔진 버전이 올라가도 **입출력 스키마 버전**을 명시해 호환을 유지한다.

---

## 2. 입력값 리스트 (엔진에 보낼 데이터)

호출 측이 **시그널 요청** 시 엔진에 넘겨야 할 payload. (실제 전송 형식은 프로토콜에 따라 JSON/Protobuf 등.)

### 2.1 필수 입력

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `request_id` | string | 요청 식별자. 로그·중복 처리용. |
| `timestamp_utc` | string (ISO 8601) | 요청 시각(UTC). |
| `market` | string | 대상 마켓. 예: `KRW-BTC`, `KRW-ETH`. |
| `mode` | string | `"entry"` \| `"exit"` \| `"both"`. 진입 판단만 / 매각 판단만 / 둘 다. |
| `candles_1h` | array | 최근 1시간봉 N개. 각 요소: `{ "t": 시각, "o": 시가, "h": 고가, "l": 저가, "c": 종가, "v": 거래량 }`. |
| `candles_4h` | array | 최근 4시간봉 N개. 형식 동일. |
| `current_price` | number | 현재가. |
| `positions` | array | 현재 보유 포지션. 각 요소: `{ "market", "quantity", "avg_entry_price", "entry_timestamp_utc" (선택, ISO 8601), "sold_tier1", "sold_tier2", "sold_tier3" (선택, bool) }`. 없으면 `[]`. |
| `balance_krw` | number | 원화 잔고. |
| `config` | object | 봇 설정. 아래 2.2 참고. |

### 2.2 설정(config) 필드

| 필드명 | 타입 | 설명 | 기본 예시 |
|--------|------|------|-----------|
| `max_positions` | int | 최대 동시 포지션 수. | 7 |
| `stop_loss_pct` | number | 손절 비율(%). | 2.5 |
| `take_profit_pct` | number | 전량 익절 비율(%). | 7.0 |
| `take_profit_tier1_pct` | number | 분할 익절 1단계(%). | 5.0 |
| `take_profit_tier2_pct` | number | 분할 익절 2단계(%). | 10.0 |
| `take_profit_tier3_pct` | number | 분할 익절 3단계(%). | 15.0 |
| `time_stop_hours` | int | 시간 손절 경과 시간(시간). | 12 |
| `max_investment_ratio` | number | 0~1. 투자 비중 상한. | 0.5 |
| `event_window_active` | bool | 미국 거시 이벤트 창 여부. true면 진입 제한 등 적용. | false |

### 2.3 선택 입력 (T2/T3 팩터 사용 시)

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `binance_prices` | object | `{ "BTCUSDT": 가격, ... }`. 바이낸스 시세. |
| `kimchi_premium_pct` | number | 김치프리미엄(%). |
| `macro_event_today` | bool | 당일 거시 이벤트 유무. |
| `exchange_net_flow_7d` | number | 거래소 순유입 7일 (선택). |
| `exchange_net_flow_30d` | number | 거래소 순유입 30일 (선택). |
| `market_regime` | string | 시장 국면. `"up"` \| `"sideways"` \| `"down"`. 진입 1차·매각 1순위(국면 하락)·7순위 참고. | (선택) |
| `market_score` | number | 국면 점수. 3차 B안(강한 추세)·시간 손절 3차 조건 등. | (선택) |

- 위 선택 입력이 없으면 엔진은 **T1만** 또는 **기본값**으로 동작한다.

## 3. 출력값 리스트 (엔진이 돌려주는 데이터)

엔진이 **시그널 요청**에 대해 반환하는 payload.

### 3.1 공통 필드

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `request_id` | string | 요청 시 보낸 `request_id` 그대로. |
| `timestamp_utc` | string (ISO 8601) | 응답 시각(UTC). |
| `schema_version` | string | 입출력 스키마 버전. 예: `"1.0"`. 호환성 확인용. |
| `status` | string | `"ok"` \| `"error"`. |
| `error_code` | string (optional) | `status == "error"` 일 때 코드. |
| `error_message` | string (optional) | 사람이 읽기 쉬운 메시지. |

### 3.2 status == "ok" 일 때

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `signal` | string | `"hold"` \| `"buy"` \| `"sell"`. 관망 / 매수 / 매도. |
| `reason_code` | string | 판단 이유 코드. 예: `"entry_1_2_3_ok"`, `"exit_stop_loss"`, `"exit_take_profit"`, `"hold_event_window"`. |
| `reason_text` | string (optional) | 이유 설명(로깅·디버깅용). |
| `market` | string | 대상 마켓. |
| `side` | string (optional) | `"buy"` \| `"sell"`. signal이 buy/sell일 때. |
| `quantity` | number (optional) | 매수/매도 수량 제안. (호출 측이 주문 실행 시 참고.) |
| `price_limit` | number (optional) | 지정가 사용 시 제안 가격. |
| `metadata` | object (optional) | 엔진 버전·점수 등 부가 정보. |

### 3.3 reason_code 예시 (참고)

- **진입**: `entry_1_2_3_ok`, `entry_1_2_3_ok_t2_filtered` 등.
- **매각**: `exit_stop_loss`, `exit_take_profit`, `exit_dead_cross`, `exit_time_stop`, `exit_market_downturn` 등.
- **보류**: `hold_event_window`, `hold_net_flow_high`, `hold_no_signal` 등.

(실제 코드값은 엔진 구현 시 이 문서에 맞춰 확정·배포.)

## 4. 엔진만 사용할 경우 연동 방법(코딩 가이드)

엔진 **바이너리만** 설치해 두고, 자체 백엔드·봇에서 **엔진을 호출하는 쪽**이 따라야 할 절차.

### 4.1 사전 준비

1. **엔진 바이너리**를 서버에 설치하고, **설정 파일** 또는 **환경 변수**로 수신 주소·포트를 지정.
2. 엔진 프로세스를 **기동**. (systemd/supervisor 등으로 상시 실행 가능.)
3. **입출력 스키마 버전** 확인: 엔진 문서 또는 `GET /version` 같은 엔드포인트에서 `schema_version` 확인.

### 4.2 호출 순서(권장)

1. **봇 시작** 시: 엔진 프로세스가 떠 있는지 확인. (health check.)
2. **주기(예: 1분)** 마다 또는 **캔들 종료 시**:
   - 현재 **시세·캔들·포지션·잔고·설정**을 수집.
   - 위 **§2 입력값 리스트**에 맞춰 JSON(또는 정한 형식)으로 구성.
   - 엔진 **시그널 API**에 POST (또는 정한 메서드).
3. 응답 **§3 출력값 리스트**에 따라:
   - `signal == "buy"` 이면: 호출 측이 **매수 주문** 실행 (업비트 API 등).
   - `signal == "sell"` 이면: **매도 주문** 실행.
   - `signal == "hold"` 이면: 아무 동작 안 함.
4. **에러 처리**: `status == "error"` 이면 `error_code`·`error_message` 로깅 후 재시도 또는 알림.

### 4.3 예제 요청(JSON)

```json
{
  "request_id": "req-20260305-001",
  "timestamp_utc": "2026-03-05T10:00:00Z",
  "market": "KRW-BTC",
  "mode": "both",
  "candles_1h": [
    { "t": "2026-03-05T09:00:00Z", "o": 95000000, "h": 95500000, "l": 94800000, "c": 95200000, "v": 1000 },
    { "t": "2026-03-05T10:00:00Z", "o": 95200000, "h": 95800000, "l": 95100000, "c": 95600000, "v": 1200 }
  ],
  "candles_4h": [],
  "current_price": 95600000,
  "positions": [],
  "balance_krw": 10000000,
  "config": {
    "max_positions": 7,
    "stop_loss_pct": 2.5,
    "take_profit_pct": 7.0,
    "take_profit_tier1_pct": 5.0,
    "take_profit_tier2_pct": 10.0,
    "take_profit_tier3_pct": 15.0,
    "time_stop_hours": 12,
    "max_investment_ratio": 0.5,
    "event_window_active": false
  },
  "market_regime": "up",
  "market_score": 0
}
```

```json
  "timestamp_utc": "2026-03-05T10:00:01Z",
```

```json

  "timestamp_utc": "2026-03-05T10:00:01Z",
  "schema_version": "1.0",
  "status": "error",

  "error_message": "candles_1h 개수 부족 (최소 24개 필요)"
```

### 4.6 코딩 시 유의사항

- **타임아웃**: 엔진 응답 타임아웃(예: 5초) 설정 후, 초과 시 재시도 또는 보류 처리.
- **재시도**: 일시적 오류(`error_code`)에 따라 재시도 횟수·간격 정책 적용.
- **스키마 버전**: `schema_version` 이 올라가면 **호환되는 필드만** 사용하거나, 마이그레이션 가이드에 따라 수정.
- **주문 실행**: 엔진은 **시그널만** 반환. 실제 주문(업비트 API 호출 등)은 **호출 측**이 구현. (엔진이 주문 API를 직접 호출하는 옵션은 별도 규격.)

### 4.7 T2/T3 필터 및 수수료·스프레드 보정

- **T2/T3 입력**(`binance_prices`, `kimchi_premium_pct`, `exchange_net_flow_7d` 등)을 넣을 때:
  - **김치프리미엄**: 업비트–바이낸스 가격 차이(%)를 그대로 넣거나, 호출 측에서 **수수료·스프레드**를 반영해 보정한 뒤 넣을 수 있다. 엔진은 받은 값을 기준으로 “과열 구간 진입 감점/보류” 등 필터만 수행한다.
  - **수수료 보정**: 업비트/바이낸스 수수료율을 호출 측이 알고 있으면, 프리미엄 계산 시 `(업비트가 - 수수료) vs (바이낸스가 + 수수료)` 형태로 보정 후 `kimchi_premium_pct`에 넣으면, 실제 수익성에 가까운 판단이 가능하다.
- **Hummingbot 등 참고**: 두 거래소 간 가격 차이·차익거래 시 “수수료·가스비”를 반영해 실제 이익이 나는지 계산하는 방식과 동일한 원칙을 적용하면 된다.

### 4.8 CCXT fetch_ohlcv → 엔진 캔들 변환

CCXT의 `fetch_ohlcv(symbol, timeframe)` 결과는 `[timestamp, open, high, low, close, volume]` 배열의 리스트이다. 엔진 입력 `candles_1h` / `candles_4h`는 `{ "t", "o", "h", "l", "c", "v" }` 객체 배열이므로, 호출 측에서 아래처럼 변환하면 된다.

- `t`: `timestamp`를 ISO 8601 문자열로 변환 (예: `datetime.utcfromtimestamp(ts/1000).strftime("%Y-%m-%dT%H:%M:%SZ")`).
- `o`, `h`, `l`, `c`, `v`: 각각 `open`, `high`, `low`, `close`, `volume` (인덱스 1~5)에 대응.

**데이터 흐름 요약**: 업비트/바이낸스 API 또는 CCXT → 1h·4h 캔들 수집 → 위 형식으로 변환 → `timestamp_utc`·`current_price`·포지션·설정과 함께 엔진 `/signal`에 POST. (상세: [트레이딩_엔진_상세_기획서.md](트레이딩_엔진_상세_기획서.md) §1·§6.)

## 5. 버전 관리

- **엔진 바이너리 버전**과 **입출력 스키마 버전**을 구분한다.
- **스키마 버전**(예: `1.0`)을 응답에 항상 넣어, 호출 측이 “이 버전 기준으로 코딩했다”고 검증할 수 있게 한다.
- 스키마를 **하위 호환** 가능하게 변경하거나, **마이그레이션 안내**를 문서로 제공한다.

---

## 6. 정리

- **입력**: §2 리스트대로 JSON(또는 정한 형식)으로 전달.
- **출력**: §3 리스트대로 `signal`·`reason_code`·수량 등 수신 후, 호출 측이 주문·DB 반영.
- **엔진만 사용할 경우**: §4 호출 순서·예제·에러 처리대로 코딩하면, **엔진 바이너리만** 갈아 끼워도 연동 유지.
- **유료 엔진** 배포 시 이 문서를 **공개 연동 규격**으로 제공하고, 엔진 구현체는 **바이너리만** 배포한다.
