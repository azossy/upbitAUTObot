## 1. 참조 외부 정보 및 신뢰도

엔진이 **참조해야 할 외부 정보**를 출처별·신뢰도 순으로 정리한다. 구현 시 API·수집 주기·폴백을 명시할 것.

### 1.1 공신력 높음 (1순위 참조)

| 구분 | 정보 내용 | 출처 | 비고 |
|------|-----------|------|------|
| **업비트 시세·캔들** | 현재가, 1h/4h 캔들, 호가, 체결 | [업비트 공식 API](https://docs.upbit.com/) | 진입·매각 핵심 데이터. Rate limit 10회/초 등 준수. |
| **바이낸스 시세** | 주요 코인 USDT 시세·등락률 | [바이낸스 공식 API](https://developers.binance.com/) | 글로벌 추세·선행 확인용. 연구자료 §12. |
| **미국 거시 일정** | FOMC, CPI, 비농업부문고용(NFP), 실업수당청구 | [Fed 공식](https://www.federalreserve.gov/), [Investing.com 경제캘린더](https://www.investing.com/economic-calendar/) 등 | 이벤트 창(진입·증액 제한)용. §2 참고. |
| **나스닥·S&P 지수** | 일봉 또는 1h 종가·등락률 | 공식 거래소·Yahoo Finance·제3자 API | BTC–나스닥 상관 높을 때 공통 이벤트 필터. 연구자료 §11. |

### 1.2 공신력 중간 (보조·필터)

| 구분 | 정보 내용 | 출처 | 비고 |
|------|-----------|------|------|
| **김치프리미엄 지수** | 업비트–바이낸스 가격 차이(%) | [CryptoQuant 한국 프리미엄](https://cryptoquant.com/), [kimchipremium.info](https://kimchipremium.info/), [금프리미엄닷컴](https://goldkimp.com) | 비정상 확대 시 과열·조정 참고. 연구자료 §12. |
| **거래소 순유입/순유출** | BTC 등 거래소 Net flow (7d·30d) | [CryptoQuant](https://cryptoquant.com/), [Glassnode](https://glassnode.com/) (유료) | 고래·대형 이동의 **추세** 반영. §3. |
| **고래 알림·대량 이동** | 100만 달러 이상 이체 방향·규모 | [Whale Alert](https://whale-alert.io/), [Lookonchain](https://lookonchain.com/), [WhaleQuant](https://whalequant.io/) | 단발보다 **기간별 순유입**으로 집계 권장. 연구자료 §9·§10. |

### 1.3 참고·선택 (노이즈 주의)

| 구분 | 정보 내용 | 출처 | 비고 |
|------|-----------|------|------|
| **Reddit·커뮤니티 심리** | r/CryptoCurrency 등 | 공개 포럼 | 정량화 어려움. 보조 참고만. 연구자료 §8. |
| **뉴스 헤드라인** | 연준·CPI·규제 뉴스 | CoinDesk, Forbes, CNBC 등 | 이벤트 일정과 병행해 “이벤트 창” 보강용. |

### 1.4 데이터 수집·갱신 권장

- **업비트**: 실시간·1h/4h 캔들 — 엔진 주기(예: 1분~5분)마다 또는 WebSocket.
- **바이낸스**: 1h·일봉 등락률 — 5~15분마다 또는 캔들 종봉 시.
- **미국 거시 일정**: 일 1회 또는 캘린더 API로 “오늘·내일” 이벤트만 조회.
- **나스닥**: 1h 또는 일봉 — 1시간 단위 갱신.
- **김치프리미엄·순유입**: 1시간~일 단위(무료 한도 고려).
- **고래 알림**: 실시간 구독 시 이벤트만 수신 후, 7d/30d **집계값**으로만 사용 권장.

## 2. 미국 경제 뉴스와 코인 가격 상관관계·엔진 반영

연구자료 §11(강박사) 요약 및 엔진 반영 방안.

### 2.1 상관관계 요약

| 지표/이벤트 | 코인(비트코인 등) 반응 | 근거·비고 |
|-------------|------------------------|-----------|
| **연준 금리 인하** | 단기 **지지** (유동성 기대). 다만 연준 **호우이시 톤** 시 하락 사례 | [17][18][19] |
| **CPI 예상보다 낮음** | BTC·나스닥 동반 **상승** | [21][23] |
| **CPI 예상보다 높음·고착** | 국채 금리↑·연준 강경론 → 코인 **부담** | [21][23] |
| **고용 강함** | 금리 유지·인상 가능성↑ → 코인 **부담** | [22] |
| **고용 약함** | 금리 인하 기대↑ → 유리할 수 있으나, **경기 둔화 우려** 시 리스크 오프 동반 하락 | [21] |
| **비트코인–나스닥 30일 상관** | **약 0.70** 등 높을 때, 미국 거시 이벤트가 주식·코인에 **동시 영향** | [23][24] |
| **실업수당청구 급증** | 경기 불안·리스크 오프 심리 확대 가능 | [21] |

### 2.2 엔진 반영 방식

- **이벤트 창(Event window)**  
  - **CPI·FOMC·NFP(비농업부문고용)·실업수당청구** 발표일 **D-1 00:00 ~ D+1 24:00 (UTC 기준)** 를 “이벤트 창”으로 정의.
  - 이벤트 창 내에서는:
    - **진입**: 신규 매수·증액 **제한** (또는 최대 비중 축소).
    - **매각**: 기존 로직(손절·데드크로스·익절·시간손절)은 **유지**. 필요 시 “이벤트 창 내 손절 폭을 -2.0%” 등으로 **일시 강화** 옵션.
  - 일정은 **외부 캘린더 API** 또는 **정적 테이블**(월별 업데이트)로 관리.

- **금리·유동성 구간 플래그 (선택)**  
  - “금리 인하 기대 상승” 구간(예: CPI 하락·연준 완화 발언 후) vs “금리 고정·인상 우려” 구간을 플래그로 두고,  
    전자에서는 진입 필터를 **완화**(예: ADX 임계 23), 후자에서는 **강화**(ADX≥27)하는 식의 **파라미터 스위칭**을 검토.

- **나스닥 동행 구간**  
  - BTC–나스닥 상관이 **임계값 이상**(예: 0.6)일 때, “미국 거시 이벤트”를 **공통 입력**으로 사용하고, 위 이벤트 창을 **나스닥 변동성 확대 구간**과 겹쳐 적용할 수 있음.

### 2.3 참조할 공식·매체

- 연준: [Federal Reserve](https://www.federalreserve.gov/), FOMC 일정.
- CPI·고용: [BLS](https://www.bls.gov/), [Investing.com Economic Calendar](https://www.investing.com/economic-calendar/).
- 연구자료 §14 참고문헌 [17]~[24].

## 3. 대형업체·고래 코인 이동 팩터

연구자료 §9·§10(김박사) 요약 및 엔진 반영.

### 3.1 시장 영향 요약

| 현상 | 해석·시장 영향 |
|------|----------------|
| **거래소 대량 유입** | 매도 의도·공급 증가 → **매도 압력·변동성 확대** [13][14] |
| **거래소 대량 유출** | 축적·유동성 감소 → **매수·홀딩 심리** 참고 [11][12] |
| **고래 유입 비율 급상승** | 소수 대형 보유자 유동성 지배 → **동시 매도 시 가격·변동성 영향 큼** [13] |
| **단기 보유자 손실 매도 + 고래 유입** | **캐피튜레이션·국면 전환** 구간 해석 [13] |
| **알트코인 대량 유입** | 유통 공급·심리 영향 (예: XRP 업비트 매도 파이프라인 [28]) |

### 3.2 엔진에 넣을 팩터 (보조 필터)

- **지표로 사용할 것**:  
  - **거래소 순유입(Net flow)** 의 **7일·30일 추이** (단일 이체가 아닌 **기간 집계**로 노이즈 감소).
- **진입 시**  
  - “해당 코인(또는 BTC) **7일 순유입**이 임계값 이상”(순유입 과다 → 매도 압력 축적)이면 **진입 감점** 또는 **진입 보류**.
  - “**순유출** 지속”(축적 구간)이면 진입 점수 **소폭 가산** (선택).
- **매각 시**  
  - “**순유입 급증**”을 “매도 압력 증가” 보조 신호로 두고, **기존 손절·데드크로스보다 우선순위는 낮게** 두고,  
    동일 2순위 이하 조건에서 “순유입 임계 초과”일 때 **손절 폭을 -2.0%** 등으로 일시 강화하는 옵션 검토.
- **데이터 소스**  
  - CryptoQuant, Glassnode 등 **API**에서 Net flow·Exchange Inflow/Outflow (7d/30d) 수치를 주기적으로 받아와 **스코어화** (예: -1~0~+1 또는 0~100).

### 3.3 업비트 전용 유동성 (챠리 §12)

- 업비트에는 **거래소 특유 매도 압력**(대량 매도 파이프라인·알고리즘 매도)이 있어, 바이낸스 선행만으로는 설명 부족 [28].
- **업비트 호가·체결량 급변** (대량 매도/매수 체결)을 **별도 모니터링**하면 “진입 자제” 또는 “손절 강화” 보조 신호로 활용 가능.  
  → 구현 시 업비트 WebSocket 체결·호가 누적으로 “이상 거래량” 감지 로직 추가 검토.

## 4. 팩터 신뢰도와 가중치 설계

각 팩터를 **신뢰도(재현성·검증 가능성)** 순으로 구분하고, **진입/매각 점수 또는 필터**에 반영하는 가중치 원칙을 정리한다.

### 4.1 신뢰도 등급 정의

| 등급 | 설명 | 예시 |
|------|------|------|
| **T1 (높음)** | 업비트·바이낸스 등 **공식 시세·캔들** 기반, 조건이 명확하고 재현 가능 | EMA 골든/데드크로스, ADX, RSI, 거래량(업비트) |
| **T2 (중간)** | **공식·제3자 공인 데이터**이지만 해석·기간에 따라 결과 차이 | 미국 거시 일정(이벤트 창), 나스닥 지수, 김치프리미엄, 거래소 순유입(7d/30d) |
| **T3 (보조)** | 유용하나 **노이즈·지연·단일 이벤트** 가능성 | 고래 단발 알림, Reddit 심리, 뉴스 헤드라인 |

### 4.2 진입 시 팩터 가중치 (권장 비율)

진입 “통과/불통과” 및 **진입 강도 점수**에 반영할 때의 상대 비중 제안. (합 = 100%)

| 팩터 | 신뢰도 | 권장 비중 | 비고 |
|------|--------|-----------|------|
| 1h·4h EMA(골든크로스·정배열) | T1 | 25% | 핵심 추세 신호. |
| ADX·거래량(평균 대비) | T1 | 25% | 추세 강도·관심도. |
| RSI·눌림목(3차) | T1 | 20% | 진입 타이밍. |
| 시장 국면(상승/횡보/하락) | T1 | 15% | 1차 필터. |
| 바이낸스 선행·김치프리미엄 | T2 | 8% | 글로벌 추세·과열 보조. |
| 미국 거시 이벤트 창 | T2 | 5% | 이벤트 창 내 진입 감점/제한. |
| 거래소 순유입(7d/30d) | T2 | 2% | 순유입 과다 시 감점. |
| 고래·온체인 (집계) | T3 | 0~2% | 선택. 단발 제외. |

- **진입 최종**: 기존처럼 **1차·2차·3차 모두 통과**가 필수.  
  T2·T3는 “통과 후 **점수 가감**” 또는 “이벤트 창 시 **진입 불가**” 같은 **보조 필터**로만 사용.

### 4.3 매각 시 팩터 우선순위 (기존 유지 + 보조)

매각은 **기존 우선순위**(국면 하락 → 손절 → 데드크로스 → 익절 → 시간손절)를 유지하고:

- **T2 보조**:  
  - 이벤트 창 내에서는 손절 **-2.0%** 등 **일시 강화** 옵션.  
  - 순유입 급증 시 동일 2순위(손절)에서 **손절 폭 소폭 강화** 옵션.
- **T3**:  
  - “고래 대량 유입 알림” 단독으로는 **매각 트리거로 쓰지 않음**.  
  - 7d/30d **순유입**이 임계 초과일 때만 위 T2와 연동해 손절 강화 검토.

### 4.4 가중치 조정 원칙

- **백테스트·페이퍼트레이딩**으로 기간별 수익률·최대낙폭·승률을 보고, T1 비중을 **먼저** 고정한 뒤 T2/T3 비중을 **소폭**만 조정.
- **과최적화(커브피팅)** 방지: T2·T3 합계는 **15% 이하**로 두고, “이벤트 창” “순유입 임계” 등 **파라미터 개수**를 최소화.

## 5. 엔진 입력·출력 및 데이터 흐름

### 5.1 입력 (주기적 수집)

| 입력 | 주기 | 출처 | 용도 |
|------|------|------|------|
| 업비트 캔들(1h, 4h) | 1봉 종료 시 또는 1~5분 | 업비트 API | EMA, ADX, RSI, 거래량, 시장 점수 |
| 업비트 현재가·호가 | 실시간 또는 1분 | 업비트 API/WebSocket | 손절·익절 1차 조건 |
| 바이낸스 시세(주요 코인) | 5~15분 | 바이낸스 API | 선행·김치프리미엄 |
| 미국 거시 일정 | 일 1회 또는 캘린더 | 외부 캘린더/테이블 | 이벤트 창 플래그 |
| 나스닥(또는 S&P) 1h/일 | 1시간 | 제3자 API | 상관·이벤트 구간 보조 |
| 김치프리미엄 | 1시간~일 | CryptoQuant 등 | 과열 구간 플래그 |
| 거래소 순유입(7d/30d) | 1시간~일 | CryptoQuant/Glassnode | 진입 감점·손절 강화 옵션 |

### 5.2 출력 (엔진 결정)

- **진입**: 1차·2차·3차 통과 + T2/T3 보조 필터 통과 시 **매수 신호** (종목·비중·분할).
- **매각**: 1~7순위 검사 순서대로 **한 가지 사유**로 **전량/분할 매도** (손절·익절·시간손절 등).
- **진입 보류**: 이벤트 창·순유입 과다·“진입 자제” 온체인 점수 등.

### 5.3 데이터 흐름 (개념)

```
[업비트 캔들/현재가] ──► [EMA·ADX·RSI·거래량·시장점수] ──► [1차·2차·3차 진입 판단]
[바이낸스 시세] ──────► [김치프리미엄·선행 여부] ──────────► [T2 보조 필터]
[미국 거시 일정] ─────► [이벤트 창 플래그] ─────────────────► [진입 제한·손절 강화 옵션]
[나스닥] ─────────────► [상관 구간 플래그] ─────────────────► [이벤트 창 보강]
[순유입 7d/30d] ─────► [진입 감점·손절 강화] ───────────────► [필터·옵션]
                              │
                              ▼
                    [최종 진입/매각/보류 신호]
```

## 6. 서버 설정 요약(외부 데이터 연동)

엔진이 외부 사이트에서 정보를 가져오기 위해 **서버에 설정해야 할 항목**을 정리한다. 가입·API 키·유무료를 표기하며, **유료는 꼭 필요할 때만** 도입하는 것을 권장한다.

### 6.1 필수·무료 위주 (서버 기본 설정)

| 사이트/서비스 | 유/무료 | 서버에 설정할 것 | 비고 |
|---------------|---------|------------------|------|
| **업비트** | **무료** | ① 업비트 **회원가입** ② [마이페이지 → API 관리](https://upbit.com/mypage/open_api_management)에서 **API Key 발급** (Access Key, Secret Key) ③ 보안 레벨 4 이상 필요. **아이디·비밀번호**는 업비트 로그인용이며, API 호출에는 **Access Key·Secret Key**만 사용. | 시세·캔들·주문 핵심. 키는 서버 `.env` 등에 저장, 노출 금지. |
| **바이낸스** | **무료** | ① 바이낸스 **회원가입** ② [API Management](https://www.binance.com/en/my/settings/api-management)에서 **API Key 생성** ③ 시세만 읽을 경우 **읽기 전용** 키로 생성. **API Key + Secret** 두 값만 서버에 설정. 아이디·비밀번호는 사이트 로그인용, API는 키만 사용. | 글로벌 시세·김치프리미엄 계산용. 주문 안 하면 읽기 전용 키 권장. |

### 6.2 무료로 쓸 수 있는 보조 데이터

| 사이트/서비스 | 유/무료 | 서버에 설정할 것 | 비고 |
|---------------|---------|------------------|------|
| **미국 거시 일정** (FOMC·CPI·NFP) | **무료** | ① **가입 불필요**인 공개 소스 사용 시: Fed·BLS 공식 페이지는 스크래핑 또는 고정 URL. ② 자동화 시 **Investing.com 경제캘린더** 등은 API가 **유료**인 경우 많음 → **정적 테이블**(월별 FOMC·CPI·NFP 일정)을 서버에 두고 주기적으로 수동/스크립트로 갱신하는 방식 권장. 별도 API 키 없이 가능. | 이벤트 창만 쓰면 월 1회 수동 입력으로도 가능. |
| **나스닥·S&P 지수** | **무료** | ① **Yahoo Finance** 등 무료 API(yfinance 등) 사용 시 **가입·API 키 없이** 사용 가능한 라이브러리 있음. ② 또는 Alpha Vantage 등 **무료 티어**(API 키 발급 후 일 25회 등 제한)로 키 1개만 서버에 설정. | 무료 한도 내에서 1시간봉/일봉만 받으면 충분. |
| **김치프리미엄** (자체 계산) | **무료** | **설정 없음**. 업비트 원화 시세 + 바이낸스 USDT 시세 + **환율**(공개 API 또는 고정값)으로 서버에서 직접 계산. 별도 사이트 가입·API 불필요. | kimchipremium.info·금프리미엄닷컴은 **참고용** 열람만 해도 됨. |

### 6.5 서버 환경 변수 예시(요약)

아래는 **무료·필수 위주**만 쓸 때의 예시다. 유료 서비스 도입 시 해당 사이트 문서에 따라 키 이름을 추가하면 된다.

```bash
# 필수 (업비트 실제 매매 시)
UPBIT_ACCESS_KEY=...
UPBIT_SECRET_KEY=...

# 바이낸스 시세 전용 (읽기 전용 키)
BINANCE_API_KEY=...
BINANCE_API_SECRET=...

# 선택: 나스닥 등 (무료 API 사용 시)
# ALPHA_VANTAGE_KEY=...   # 무료 티어 25회/일 등

# 선택: 미국 거시 이벤트 (정적 테이블 사용 시에는 불필요)
# INVESTING_CALENDAR_API_KEY=...

# 유료·꼭 필요할 때만
# CRYPTOQUANT_API_KEY=...
# GLASSNODE_API_KEY=...
# WHALE_ALERT_API_KEY=...
```

- **아이디·비밀번호**: 업비트·바이낸스는 **웹 로그인용**만 쓰고, **API 호출에는 사용하지 않음**. 서버에는 **API Key + Secret** 만 두면 된다.
- **유료**: 순유입·고래·경제캘린더 자동화를 **모두 생략**해도, 업비트+바이낸스+정적 이벤트 테이블만으로 **핵심 진입·매각·이벤트 창** 구현 가능. 유료는 **백테스트·운영 결과를 보고 꼭 필요하다고 판단될 때만** 도입하는 것을 권장한다.

## 7. 엔진 분리·교체 가능 구조(바이너리 분리)

**방침(반드시 준수)**: 트레이딩 엔진은 **개발·배포 시 항상 바이너리로만** 제공한다. 소스코드는 공개하지 않고, **별도 프로세스(컴파일된 실행 파일)**로 분리해 **엔진만 갈아 끼울 수 있게** 한다. 문서(기획서·입출력 명세·연동 가이드)는 공개 가능하며, **유료 엔진**으로 고도화·판매할 때도 이 원칙을 따른다.

### 7.1 구조 개요

```
[공개 백엔드 (FastAPI)]  ←→  HTTP/gRPC/소켓  ←→  [트레이딩 엔진 (별도 바이너리)]
       ↑                                                    ↑
  저장소에 소스 공개 가능                            소스 비공개, 바이너리만 배포
```

- **백엔드**: 사용자 인증·봇 on/off·포지션·잔고·설정 등 **REST API** 담당. 트레이딩 **판단·주문 실행**은 하지 않고, **엔진 프로세스**에 요청을 보내고 결과만 받아서 DB·앱에 반영.
- **엔진**: Go / Rust / C++ 등으로 작성 후 **단일 실행 파일**로 빌드. 시세·팩터 수집·진입·매각·손절 로직 전부 포함. **소스는 비공개**, **바이너리만** 서버에 설치·실행.

### 7.2 통신 방식(인터페이스)

엔진을 나중에 **갈아 끼우려면** 백엔드–엔진 간 **요청/응답 규격**을 고정해 두는 것이 좋다. 아래는 권장 형태이며, 구현 시 프로토콜을 선택하면 된다.

| 방식 | 장점 | 비고 |
|------|------|------|
| **HTTP (REST)** | 구현 단순, 디버깅 쉬움, 언어 무관 | 엔진이 내부에 HTTP 서버 띄움. 백엔드는 `http://127.0.0.1:엔진포트/...` 호출. |
| **gRPC** | 타입·스키마 명확, 스트리밍·다중 호출에 유리 | .proto 정의만 공개하고, 엔진 구현체는 비공개. |
| **TCP 소켓(JSON 라인)** | 가벼움, 오버헤드 작음 | 한 줄에 JSON 하나. 백엔드–엔진이 같은 호스트 전제. |

- **엔진이 제공하는 것(예시)**  
  - **시그널 요청**: “현재 시세·포지션·설정”을 받고 → **매수/매도/관망** 등 **시그널 + 이유 코드** 반환.  
  - **주문 실행(선택)**: 엔진이 직접 업비트 API를 호출할 수도 있고, “이 주문 실행해라”만 백엔드에 넘기고 백엔드가 주문 API를 호출할 수도 있음. (보안·키 관리 정책에 따라 결정.)

- **백엔드가 할 일**  
  - 엔진 프로세스 **기동/종료** (봇 시작/정지 시).  
  - 주기적으로(또는 이벤트로) 엔진에 **시그널 요청**을 보내고, 응답에 따라 **DB 업데이트·주문 API 호출** 등만 수행.

### 7.3 배포·갱신

- **엔진 바이너리**: 공개 저장소에 올리지 않고, **비공개 저장소·내부 배포 채널**에서만 관리.  
- **문서**: 기획서(본 문서), **엔진 호출 규격(URL·payload·응답 형식)** 또는 **.proto** 등은 GitHub 등에 공개해도 됨.  
- **엔진만 교체**: 새 버전 바이너리로 교체 후 프로세스만 재시작하면 되고, **백엔드·앱 코드는 수정하지 않아도** 됨(인터페이스가 같다면).

### 7.4 입출력 명세·엔진만 사용 시 연동 가이드

- **입력값 리스트·출력값 리스트** 및 **엔진만 사용할 경우 어떻게 코딩해야 하는지**는 별도 문서에 정의해 공개한다.
- **문서**: [트레이딩_엔진_입출력_연동_가이드.md](트레이딩_엔진_입출력_연동_가이드.md)  
  - 엔진에 보낼 **입력 필드 리스트**(필수/선택).  
  - 엔진이 돌려주는 **출력 필드 리스트**(시그널·이유코드·에러).  
  - **엔진만 사용할 때** 호출 순서·예제 요청/응답·에러 처리 등 **연동(코딩) 방법**.
- 유료 엔진 배포 시 위 문서를 **연동 규격**으로 제공하고, 구현체는 **바이너리만** 배포한다.

### 7.5 정리

- **엔진 개발 시**: **바이너리로 빌드·배포**하는 것을 기본으로 한다.  
- **문서 공개**: 가능. (기획서·입출력 명세·연동 가이드)  
- **엔진 소스 공개**: 하지 않음.  
- **엔진 형태**: **별도 프로세스(컴파일된 바이너리)**.  
- **갈아 끼우기**: 동일한 **입출력 규격**만 지키면, 엔진 바이너리만 교체해서 사용 가능.  
- **유료화**: 엔진을 고급화해 유료로 판매할 때도, 배포는 **바이너리 + 입출력·연동 문서**로 한다.
